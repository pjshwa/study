Static and Dynamic Priority

  • Static Priority

    • Determines the length of a time slice

  • Dynamic Priority

    • Processes which sleep a lot get a higher dynamic priority

 

Invoking the schedule() function

O(1) Scheduler Problems

  • Oddities in time slice-based priority (lack of fairness)

    • Different switching rate

      • Slow response time / Low throughput: Choose one!

    • Unfair CPU allocation

      • Compare by percentage:

        • Pri=120(100ms), Pri=121(95ms) [CPU time difference 5%]
        • Pri=138(10ms), Pri=139(5ms) [CPU time difference 50%]

  • (Possible) Starvation of tasks

  • Code that implements dynamic priority is largely heuristic

 

CFS (Completely Fair Scheduler)

  • From Linux 2.6.23~
  • Goal: every process gets its fair CPU share
  • Use weight for fairness
  • Use Red-Black tree-based runqueue ( complexity)

 

Scheduling Class (1)

  • Linux scheduler has been made modular from 2.6.23

    • Modularity is called scheduling classes
    • Able to implement different scheduling policies in a modular way

 

Scheduling Class (2)

  • Scheduling classes are managed in a singly linked list.

 

Scheduling Class (3)

  • *sched_class inside your PCB (task_struct)

 

CFS: Concept

  • Provide each task CPU time proportional to its weight (?)

 

CFS: Task Weight

  • "nice" to "weight" mapping

    • 1.25 times weight for each nice level
    • 1.25 times approximately represents 10% increase/decrease of CPU usage (Don't ask why)

 

CFS: Virtual Runtime

  • Account for "how long a process has run" and "how much longer it ought to run"

  • Small virtual runtime: the task has received less CPU time than what it should have received

    • = weight of nice value 0
    • = weight of task (The only parameter(variable))

  • Scheduling decision: choose the smallest virtual runtime

    • Means that the process is urgent
