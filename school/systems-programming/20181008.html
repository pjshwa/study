O(1) Scheduler Problems

  • led to CFS, which introduces a new concept: weights

 

CFS: Task Weight

  • "nice" to "weight" mappings

 

CFS: Virtual Runtime

  • Task's cumulative executed runtime inversely scaled by its weight.

 

CFS: Time Slice

  • Proportional to the task's weight

 

CFS: Red-Black(RB) Tree

  • Task selection with RB tree

    • All runnable tasks are sorted by vruntime
    • Self-balancing
    • Operations in time
    • Simply take the leftmost node!

 

CFS: Scheduler Entity structure

  • Process accounting

    • Uses the scheduler entity structure, struct sched_entity to keep track of process accounting.
    • Embedded in struct task_struct as a member variable named se

 

Run Queue Structure: Linux 2.6.23~

  • Each CPU owns its run queue (one run queue for each CPU)

    • Run queue is implemented using the struct rq data structure
    • Priority 0~99: still SCHED_FIFO or SCHED_RR (hasn't been changed.)
    • Priority 100~: Implemented in CFS RB-tree.

Multiprocessor Scheduling in Linux (1)

  • Load balancing: distribute workload evenly across CPUs.

    • To get optimal resource utilization, maximize throughput, and minimize response time

  • Note: load balancing overhead

    • Information collection

      • Collect runtime information

    • Task migration

      • Scheduling cost: Update kernel data structure, context switch, ..
      • Cold cache effect: Previous cache becomes unusable

 

Multiprocessor Scheduling in Linux (2)

  • Linux CFS scheduler attempts to evenly distribute load across multiple CPUs

  • Principles

    • Prevent CPUs from being idle
    • Keep the difference in numbers of ready tasks on all CPUs as small as possible.

 

Scheduling Domains and Scheduling Groups

  • Scheduling Domain

    • Load balancing takes place "within a scheduling domain"
    • Each scheduling domain must have one or more scheduling groups

  • Scheduling Group

    • Each scheduling group contains one or more CPUs
    • Load balancing takes place "between scheduling groups"

 

Triggering Load Balancing in Linux

  • Time-driven load balancing (triggered by timer interrupt)

    • Load balancing code trigger_load_balance() is invoked periodically with scheduler_tick()
    • If needed, perform load balancing via run_rebalance_domains()

  • Event-driven load balancing (triggered by scheduling events)

    • When a task is newly created, or woken up via fork(), exec(), wakeup()

      • Select the least loaded group in the current domain, and move the task there

    • When the CPU becomes newly idle

      • Select the most loaded group, and move a task from it to the newly idle CPU

Kernel-Mode Preemption (1)

  • Note: User-Mode Preemption

    • When the kernel is about to return to user-space
    • If TIF_NEED_RESCHED is set, the scheduler is invoked to select a new process to execute.

  • Kernel-Mode Preemption

    • Since kernel version 2.6
    • A process can be preempted executing in kernel mode
    • No need to delay preemption until the current process returns to user mode.

 

Kernel-Mode Preemption (2)

  • Processes can be preempted in kernel mode as long as they do not hold a lock

    • Locks used as markers of non-preemptability

  • preempt_count: count of locks acquired

    • preempt_count == 0 means the process is preemptable
